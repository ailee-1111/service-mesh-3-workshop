# 모듈 1: 서비스 메시 3 (OSSM v3) 아키텍처 살펴보기

오픈시프트 서비스 메시 3.0(OSSM v3)은 독자 엔진 기반에서 완벽히 탈피하여, 글로벌 업스트림 표준인 **Istio(이스티오) 아키텍처 및 Sail Operator** 기반으로 대통합을 이룬 역사적인 메이저 릴리즈입니다.

본 랩에서는 서비스 메시 3.0의 핵심 아키텍처 제어 모델과 주요 구성 요소를 이해하고, 실제 운영 환경에 배포된 제어부 컴포넌트들을 원클릭 터미널 명령을 통해 직접 관찰하는 실습을 진행합니다.

---

## 🏗️ 1. 서비스 메시 3.0 핵심 아키텍처 비교

오픈시프트 서비스 메시 3.0은 복잡하고 다소 무겁던 2.x 버전에 비해 구조가 극도로 단순화되고 제어 표준성이 크게 높아졌습니다.

| 아키텍처 구분 | 기존 버전 (OSSM v2) | 최신 버전 (OSSM v3) |
| :--- | :--- | :--- |
| **핵심 업스트림** | Maistra (이스티오 포크 버전) | **Istio.io 표준 (순수 업스트림)** |
| **라이프사이클 엔진** | Maistra Service Mesh Operator | **Sail Operator (이스티오 에코시스템 표준)** |
| **핵심 설정 리소스(CRD)**| `ServiceMeshControlPlane` (SMCP) | **`Istio` (단일 간결 명세)** |
| **소속 네임스페이스 관리**| `ServiceMeshMemberRoll` (SMMR) | **표준 네임스페이스 레이블 주입 (`istio-injection`)** |
| **제어부(Control Plane)** | 복잡한 다중 컨트롤 포드 (Pilot, 등) | **`Istiod` (단일 경량화 데몬 통합)** |
| **옵저버빌리티 연동** | SMCP 내부에 번들링되어 통제됨 | **독립형 Operator (Kiali, Prometheus 등)** |

---

## 🧩 2. OSSM v3 컨트롤 플레인의 3대 부품

OSSM v3가 안정적인 가상 네트워크 및 보안을 구성하기 위해 협업하는 3대 컴포넌트입니다:

1. **Sail Operator (관리자):** 
   * 이스티오 컨트롤 플레인의 설치, 업그레이드, 리비전(Revision) 기반 카나리 업그레이드를 완벽하게 통제하는 수명 관리 엔진입니다.
2. **Istiod (뇌):**
   * 서비스 디스커버리, xDS 트래픽 경로 분배, mTLS 인증서 자동 관리 등을 단 하나의 초경량 컨테이너 안에서 통합 수행하는 서비스 메시의 "뇌"입니다.
3. **Istio CNI Plugin (발/네트워크):**
   * 오픈시프트의 까다로운 멀티테넌트 보안 규정을 준수하면서, 일반 애플리케이션 포드에 권한 상승(Privileged)을 부여하지 않고도 트래픽을 가상 프록시(Envoy)로 부드럽게 가로채주는 특화 네트워크 드라이버입니다.

## 💻 3. [실습] 클러스터 컴포넌트 실시간 진단

자, 이제 오른쪽 화면에 열려있는 주피터 가상 터미널 환경을 이용해, 실제로 클러스터 상에 가동 중인 서비스 메시 3.0 엔진들의 상태를 원클릭으로 직접 점검해 보겠습니다.

> 💡 **안내:** 아래 회색 코드 블록을 **마우스로 직접 클릭**해 보세요. 오른쪽 터미널에서 명령이 직접 타이핑되며 자동 실행됩니다!

### 🟢 실습 1. Sail Operator 기동 상태 및 설치 확인
먼저, 서비스 메시 3.0의 라이프사이클 관리를 전담하는 **Sail Operator** 포드가 정상적으로 준비(`Running`)되어 있는지 확인합니다.

아래 명령어를 클릭하여 첫 번째 터미널(위쪽)에서 실행해 보세요.
```execute
oc get pods -n openshift-operators -l app.kubernetes.io/name=sail-operator
```

### 🟢 실습 2. 서비스 메시 3.0 설정 명세(CRD) 검증
새롭게 제공되는 단일 표준 리소스 명세인 `Istio` 커스텀 리소스 정의(CRD)가 클러스터에 정상 등록되어 있는지 조회합니다.
```execute
oc get crd | grep istio.io
```

---

## 💻 4. [실습] CNI 드라이버 가동 상태 점검 (터미널 2 활용)

이번에는 아래쪽 터미널(Terminal 2)을 활용해, 각 노드마다 배포되어 네트워킹 가로채기를 조율하는 **Istio CNI 플러그인**의 가동 및 배포 스펙(`DaemonSet`)을 검증해 보겠습니다.

> 💡 **안내:** 아래 명령어는 **두 번째 터미널(아래쪽)**로 다이렉트 주입되어 구동됩니다! 클릭해 보세요.

### 🔵 실습 3. CNI 데몬셋 가동 인프라 조회
각 워커 노드마다 1개씩 복제 배치되어 무점검 가상 네트워킹을 구현하는 CNI 파드들의 전체 개수 및 헬스 체크 상태를 출력합니다.
```execute-2
oc get daemonset -n openshift-operators
```

### 🔵 실습 4. 현재 네임스페이스 확인 및 전환
실습을 완료한 후, 앞으로 우리가 개별 실습을 배정받아 꾸려나갈 유저 고유 네임스페이스인 `kafka-%username%` 혹은 `service-mesh-%username%` 영역으로 이동하는 명령어를 테스트해 봅니다.
```execute-2
oc project %username%
```
