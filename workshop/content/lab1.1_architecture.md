# 모듈 1: OpenShift Service Mesh 3 아키텍처 탐구

레드햇 오픈시프트 서비스 메시 3(OSSM v3)은 글로벌 업스트림 표준인 Istio 프로젝트와 이를 전담 관리하는 Sail Operator를 핵심 아키텍처 엔진으로 채택하여, 플랫폼의 이식성과 운영 표준성을 극대화한 버전입니다. 

본 실습에서는 제공된 공식 교육 정보를 바탕으로 서비스 메시 3의 제어 및 데이터 평면 구조, 핵심 커스텀 리소스(CRD), 그리고 타 플랫폼과의 연동 설계를 학습하고 이를 클러스터에서 실시간 검증합니다.

---

## 1. 제어 평면(Control Plane)과 데이터 평면(Data Plane)

서비스 메시의 아키텍처는 논리적으로 제어 평면과 데이터 평면으로 엄격히 분리됩니다.

* **제어 평면 (Control Plane - 관리 센터):**
  * 데이터 평면을 통과하는 실제 네트워크 패킷을 직접 수정하거나 관여하지 않습니다.
  * 데이터 평면에 분산된 프록시들이 일관된 정책을 강제하고 텔레메트리 데이터를 수집할 수 있도록 라우팅 정책 및 보안 설정을 구성하고 안전하게 배포하는 역할을 담당합니다.
  * 서비스 메시 3에서는 기존 버전의 여러 분산 컴포넌트들이 단일 경량화 바이너리인 **istiod**로 완전히 통합되어 관리가 단순화되었습니다.

* **데이터 평면 (Data Plane - 지능형 프록시):**
  * 애플리케이션 컨테이너와 나란히 배포되어 마이크로서비스 간의 모든 네트워크 통신을 투명하게 가로채는 지능형 프록시(Envoy)들로 구성됩니다.
  * 제어 평면의 지침에 따라 동적 서비스 디스커버리, 로드 밸런싱, 암호화 통신, 서킷 브레이킹, 트래픽 분할 등의 실질적인 네트워크 처리를 수행합니다.

---

## 2. Sail Operator와 핵심 Istio CRDs

오픈시프트 서비스 메시에서 **Sail Operator**는 컨트롤 플레인의 라이프사이클을 전담 관리하며, 다음과 같은 핵심 커스텀 리소스(CR)들을 조율합니다.

### Istio
클러스터 전역에서 이스티오 컨트롤 플레인을 배포하고 관리하는 가장 상위 수준의 리소스입니다.

### IstioRevision
컨트롤 플레인의 특정 "리비전(Revision)"을 정의합니다. 리비전 이름을 분리해 정의함으로써 단일 클러스터 내에 독립된 다중 버전의 컨트롤 플레인을 동시에 가동할 수 있습니다. 이를 통해 구버전에서 신버전으로 워크로드를 안전하게 점진적으로 마이그레이션하는 카나리(Canary) 스타일의 제어부 업그레이드가 가능해집니다.
개별 네임스페이스에 `istio.io/rev=<revision-name>` 레이블을 지정하여 특정 제어부와 연동되도록 매핑합니다.

### Istio CNI Plugin
오픈시프트의 다중 사용자 보안 정책을 완벽하게 만족시키기 위해, 컨테이너 네트워크 인터페이스(CNI) 플러그인이 제어부와 격리되어 모든 클러스터 노드에 분배기 형태로 가동됩니다.
일반적으로 사이드카 인젝션 시 네트워크 포워딩 규칙 작성을 위해 필요했던 권한 상승(Privileged) init 컨테이너의 필요성을 완전히 제거하여, 기업용 보안 가이드라인을 완전하게 지킬 수 있도록 지원합니다.

## 3. 데이터 평면 배포 모델 비교: Sidecar vs Ambient

서비스 메시 3은 서비스 메시 데이터 평면 구성을 위해 전통적인 사이드카(Sidecar) 모델과 더불어 사이드카리스(Sidecar-less) 아키텍처인 **엠비언트(Ambient) 모드**를 모두 제공합니다.

* **사이드카 모드 (Sidecar Mode - 전통적 방식):**
  * 각 애플리케이션 포드 내부에 Envoy 프록시가 주입되어 모든 트래픽을 가로챕니다.
  * 이 방식은 확실한 트래픽 통제를 제공하지만 포드당 리소스 오버헤드가 발생하고 대규모 스케일 아웃 시 제어 복잡도가 증가하는 한계가 있습니다.

* **엠비언트 모드 (Ambient Mode - 차세대 방식):**
  * 포드 내부의 프록시를 완전히 제거하고, 네트워크 기능을 계층화하여 노드 및 네임스페이스 단위로 프록시를 공유합니다.
  * **L4 보안 계층 (ztunnel):** 노드 단위로 가동되는 경량 에이전트인 `ztunnel`이 mTLS 암호화 통신과 L4 수준의 인증 정책을 가볍게 수행하여 오버헤드를 극적으로 최소화합니다.
  * **L7 가상 라우팅 계층 (waypoint proxies):** L7 수준의 정교한 라우팅 제어나 서킷 브레이킹이 필수적인 네임스페이스에만 Envoy 기반의 `waypoint` 프록시를 별도로 두어 필요한 시점에만 L7 처리를 집중 수행합니다.

## 4. 플랫폼 연동 구조 (Integrations)

오픈시프트 서비스 메시 3은 플랫폼의 관찰 가능성, 보안, 배포 제어를 최적화하기 위해 다음과 같은 오픈시프트 기본 플랫폼 인프라 자원들과 유기적으로 통합됩니다.

* **OpenShift Monitoring (Prometheus & Kiali):**
  * 내장된 프로메테우스 스택을 사용해 메시 내부 메트릭을 수집하고, Kiali 콘솔 플러그인을 통해 실시간 위상도(Topology) 및 서비스 통신 헬스 체크 상태를 시각화합니다.
* **OpenTelemetry & Tempo (분산 추적):**
  * 업계 표준 수집기인 오픈텔레메트리(OpenTelemetry) 인프라 및 그라파나 템포(Tempo)를 조합하여, 복잡한 트랜잭션의 추적 경로를 정밀하게 분석합니다.
* **OpenShift Cert Manager (인증서 자동 관리):**
  * 자가 서명 인증서 대신 사내 신뢰 공인 인증기관(CA)과 연계될 수 있도록 Cert Manager를 통해 서비스 메시 전용 TLS 인증서 발급 및 라이프사이클 갱신 업무를 안전하게 자동화합니다.
* **Argo Rollouts (고급 Canary 배포):**
  * 가상 라우팅 규칙(`VirtualService`) 제어를 Argo 배포 파이프라인과 연계하여 블루-그린, 점진적 가중치 트래픽 이식 등 고도의 지능형 릴리즈를 실현합니다.

## 💻 5. [실습] 컨트롤 플레인 엔진 상태 진단

오른쪽 화면에 열려있는 첫 번째 터미널(Terminal 1)과 두 번째 터미널(Terminal 2)을 활용해, 공식 아키텍처 교재 명세에 따라 현재 클러스터에 정상 설치 및 가동 중인 서비스 메시 3 엔진들을 확인해 보겠습니다.

### 🟢 실습 1. Sail Operator 및 CNI 제어부 포드 점검
현재 클러스터의 오퍼레이터 영역에서 수명 주기를 전담 관리하는 Sail Operator 포드가 성공적으로 가동되고 있는지 확인합니다.

아래 명령어를 마우스로 클릭하여 실행해 보세요.
```execute
oc get pods -n openshift-operators -l app.kubernetes.io/name=sail-operator
```

### 🟢 실습 2. 노드별 CNI 드라이버 데몬셋 배포 확인
SFC 및 컨테이너 권한 상승 없이 안전하게 가상 방화벽(Netfilter) 규칙을 생성 및 매핑해 주는 CNI 플러그인이 각 물리 노드 단위로 가동 중인지 데몬셋을 조회합니다.
```execute
oc get daemonset -n openshift-operators
```

### 🟢 실습 3. 이스티오 컨트롤 플레인 설치 제어자(CRD) 검증
새롭게 제공되는 단일 표준 최상위 배포 지침인 `istios.sailoperator.io` 리소스 명세 정의서가 클러스터에 안전하게 등록되어 작동 준비를 마쳤는지 검출합니다.
```execute-2
oc get crd | grep sailoperator.io
```

### 🟢 실습 4. 자신의 실습 네임스페이스 환경 전환
실습 진행을 위해 자신에게 격리 배정된 네임스페이스(`service-mesh-%username%`) 영역으로 이동하여 정상 작동 여부를 점검합니다.
```execute-2
oc project %username%
```
